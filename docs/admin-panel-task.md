# Завдання: Адмін-панель для управління сайтом

## Мета

Розширити наявну адмінку (`/admin`) до повноцінної панелі управління:

1. Редагування контенту **головної сторінки** (описи компаній, зображення, посилання на кейси, таймлайн досвіду, "Про мене").
2. **Управління кейсами** — створення, редагування, публікація/приховування кейсів на базі наявної блочної системи (`Block` union з `eschool-2`).
3. **Автопереклад** UA → EN при збереженні (Claude API), з можливістю ручного правлення EN.
4. **Автоконвертація медіа** при завантаженні: зображення → WebP, відео → WebM.

## Поточний стан (що вже є)

- `/admin` (`src/app/admin/page.tsx`) — редагує описи + imageRows компаній для uk/en.
- `POST /api/admin/save` — перевіряє пароль (`ADMIN_PASSWORD`), валідує контент і комітить `src/data/content.json` у GitHub через API (`GITHUB_TOKEN`) → Vercel автоматично передеплоює (~1 хв). **Цей підхід "git як CMS" зберігаємо і розширюємо.**
- Контент зараз у трьох місцях:
  - `src/data/content.json` — редаговане (описи + зображення компаній);
  - `src/lib/translations.ts` — хардкод: компанії, лейбли навбара/футера, старі кейси (`CaseBlock`);
  - `src/app/[locale]/page.tsx` — хардкод: "Про мене", таймлайн досвіду, метадані;
  - `src/app/[locale]/cases/eschool-2/page.tsx` — хардкод: блоки нового кейсу (`blocksUk` / `blocksEn`).
- Нова блочна система кейсів описана в `CLAUDE.md` (розділ "Case page block system"): `meta`, `dark-section`, `figures`, `dark-slider`, `light-section`, `white-section`, `proposal`, `quotes`, `bullets-card`, `before-after`, `img`.

## Архітектура

### Зберігання контенту — git як CMS (без бази даних)

Весь редагований контент — JSON-файли в репозиторії. Збереження в адмінці = коміт через GitHub API (як зараз) → автодеплой Vercel. Плюси: версіонування, безкоштовно, статичний сайт лишається статичним.

Нова структура даних:

```
src/data/
  home.json          # головна: hero, компанії (опис, imageRows, caseUrl, ...), "про мене", таймлайн
  cases/
    eschool-2.json   # один файл = один кейс
    <slug>.json
```

Кожен файл містить обидві локалі: `{ "uk": {...}, "en": {...} }` + спільні поля (slug, status, медіа-шляхи — вони однакові для обох мов).

### Схема кейсу

```jsonc
{
  "slug": "eschool-2",
  "status": "published",        // published | draft
  "company": "eschool",          // прив'язка до компанії на головній
  "uk": {
    "eyebrow": "Кейс. Єдина школа",
    "title": "...",
    "steps": [{ "label": "Контекст", "href": "#context" }],
    "blocks": [ /* Block[] — та сама структура, що в eschool-2/page.tsx */ ]
  },
  "en": { /* те саме, заповнюється автоперекладом */ }
}
```

- Тип `Block` виноситься з `eschool-2/page.tsx` у `src/lib/case-blocks.ts` (спільний для сторінки, адмінки та валідації API).
- Рендеринг: новий динамічний маршрут `src/app/[locale]/cases/[slug]/page.tsx` для нових кейсів читає JSON і рендерить блоки тим самим кодом, що зараз в `eschool-2`. `generateStaticParams` — зі списку файлів у `src/data/cases/`. Кейси зі `status: "draft"` не рендеряться (404) і не показуються на головній.
- **Старі кейси** (`snov`, `westudy`, `eschool` з `translations.ts`) не чіпаємо — вони живуть за своїми маршрутами як зараз. Конфлікт маршрутів вирішити пріоритетом статичних сегментів (у Next статичний маршрут перемагає динамічний). Міграція старих кейсів у нову систему — окремий етап, не в цьому завданні.

### Головна сторінка

- Дані компаній, "Про мене" і таймлайн переїжджають з `translations.ts` / `page.tsx` у `src/data/home.json` (обидві локалі).
- Посилання "Дивитись кейс" (`caseUrl`) в адмінці — це **вибір кейсу зі списку** (dropdown опублікованих кейсів + старі кейси + "немає / готується"), а не текстове поле. При приховуванні кейсу адмінка попереджає, що на нього є посилання з головної.

### Медіа-пайплайн

Проблема: body-ліміт Vercel Functions — 4.5 МБ, відео через API-роут не пролізе. Тому:

- **Зображення** (`png/jpg/webp/gif`): завантаження через API-роут `POST /api/admin/media` → конвертація у WebP на сервері через `sharp` (quality 85, як у CLAUDE.md) → коміт у `public/images/works/<case>/` через GitHub API разом із контентом. SVG не конвертується.
- **Відео** (`mp4/mov`): конвертація у WebM **на клієнті** через `ffmpeg.wasm` прямо в адмінці (прогрес-бар), потім завантаження вже готового `.webm`. Якщо файл після конвертації > 4 МБ — завантажувати через GitHub API чанками не можна, тому:
  - варіант А (рекомендований): відео вантажити у **Vercel Blob** (client upload, без ліміту 4.5 МБ), у контенті зберігати blob-URL; `next.config` → `remotePatterns` для blob-домену;
  - варіант Б: тримати все в репо, але обмежити відео ~15–20 МБ і комітити через GitHub API з бекенда (base64, ліміт contents API — 100 МБ, але функція має встигнути за таймаут).
- В адмінці для кожного блоку з зображенням: drag&drop зона + прев'ю + автозаповнення `src` після завантаження. Ratio (`1144/640`) визначати автоматично з розмірів файлу, з можливістю правки.

### Автопереклад

- Кнопка **"Перекласти на EN"** (на рівні кейсу та на рівні головної) + автопропозиція при збереженні, якщо EN порожній або UA змінився.
- API-роут `POST /api/admin/translate`: бере UA-структуру, віддає EN-структуру. Модель — `claude-sonnet-5` (через AI Gateway або Anthropic SDK, ключ `ANTHROPIC_API_KEY` / `AI_GATEWAY_API_KEY` в env).
- Перекладаються **тільки текстові поля** (title, statement, caption, items[].text, author...), структура блоків, шляхи до зображень, ratio, іконки — копіюються як є. Реалізація: обхід структури по білому списку текстових полів, а не "переклади весь JSON" (щоб модель не ламала структуру).
- EN після перекладу редагований вручну; повторний переклад питає підтвердження перед перезаписом.
- Системний промпт перекладу: продуктовий/дизайнерський тон, зберігати терміни (retention, onboarding, feedback), не перекладати назви продуктів.

### Адмінка (UI)

Маршрути під `src/app/admin/`:

- `/admin` — дашборд: секції "Головна сторінка" і "Кейси" (список зі статусами).
- `/admin/home` — редактор головної: по компаніях (опис, зображення, caseUrl, роль, роки, інструменти, ачівменти), "Про мене", таймлайн. Перемикач UA/EN.
- `/admin/cases` — список кейсів: створити, дублювати, draft/published.
- `/admin/cases/[slug]` — редактор кейсу: список блоків з можливістю додати (меню з 10 типів), видалити, перетягнути (порядок), розгорнути/згорнути. Кожен тип блоку — своя форма (поля відповідають структурам з CLAUDE.md). Прев'ю не робимо в MVP — після деплою можна дивитись на сайті; чернетку можна дивитись локально.
- Стиль адмінки — простий, у наявній стилістиці (як поточний `/admin`), без сторонніх UI-бібліотек, крім уже підключених.

### Авторизація

Замінити поточну "перевірку пароля при кожному save" на сесію:

- `POST /api/admin/login` — перевіряє `ADMIN_PASSWORD` (timing-safe, як зараз), ставить підписаний httpOnly cookie (HMAC, секрет = `ADMIN_SESSION_SECRET`, TTL 7 днів).
- `proxy.ts` (Next 16: колишній middleware) захищає `/admin/*` і `/api/admin/*` (крім login) — без cookie редірект на форму логіну.

### API-роути (всі під захистом сесії)

- `POST /api/admin/login`
- `POST /api/admin/save` — розширити: приймає `home.json` та/або файли кейсів, валідує за схемами (типізована валідація Block union), комітить одним комітом усі змінені файли (git tree API, а не по-файлово contents API — щоб один деплой на одне збереження).
- `POST /api/admin/media` — прийом зображень, sharp → webp, коміт (або повернення файлу для включення у спільний коміт при save).
- `POST /api/admin/translate` — переклад UA → EN.

### Env-змінні (Vercel)

- `ADMIN_PASSWORD` (є), `GITHUB_TOKEN` (є), `ADMIN_SESSION_SECRET` (новий), `ANTHROPIC_API_KEY` або `AI_GATEWAY_API_KEY` (новий), `BLOB_READ_WRITE_TOKEN` (якщо варіант А для відео).

## Етапи

1. **Дані**: винести тип `Block` у `src/lib/case-blocks.ts`; створити `src/data/home.json` і `src/data/cases/eschool-2.json`, наповнивши їх поточним контентом з `translations.ts` / `page.tsx`; переключити головну та `eschool-2` на читання з JSON. Сайт має виглядати ідентично (перевірити візуально до/після). Окремо: секція "Про мене" зараз має незалежні UA/EN тексти — при міграції звести їх до єдиного UA-тексту (взяти UA-версію за основу) і прогнати через автопереклад, щоб EN надалі був похідним, а не редагувався окремо.
2. **Auth**: login-роут + cookie + захист `/admin/*`.
3. **Адмінка головної**: `/admin/home` + розширений save (multi-file commit).
4. **Адмінка кейсів**: список, редактор блоків, динамічний маршрут `cases/[slug]` для нових кейсів, зв'язок caseUrl ↔ кейси.
5. **Медіа**: завантаження зображень (sharp → webp), відео (ffmpeg.wasm → webm + Blob).
6. **Автопереклад**: translate-роут + кнопки в UI.

Кожен етап — окремий коміт(и), сайт робочий після кожного етапу.

## Важливо для виконавця

- **Next.js тут нестандартний** — перед кодом читати `node_modules/next/dist/docs/` (див. `AGENTS.md`). Зокрема перевірити: proxy/middleware, пріоритет статичних vs динамічних маршрутів, читання JSON у server components.
- Дотримуватись стилів з `CLAUDE.md` (кольори — тільки CSS-змінні, блочна система — точно за специфікацією).
- Старий кейс `cases/eschool` і старі кейси з `translations.ts` — не чіпати.
- `content.json` після міграції на `home.json` — видалити разом зі старим кодом merge в `[locale]/page.tsx`.

## Рішення по відкритих питаннях

1. **Відео**: варіант А — **Vercel Blob**. Відео вантажиться клієнтським upload напряму в Blob (обхід ліміту 4.5 МБ), у контенті зберігається blob-URL. `next.config` **не потребує** `remotePatterns` — відео рендериться нативним `<video>`, а не через `next/image`. Потрібен `BLOB_READ_WRITE_TOKEN`.
2. **Деплой ~1 хв на кожне збереження** — прийнятно, лишаємо як є.
3. **Секція "Про мене"**: зараз UA і EN — незалежні, вручну написані тексти (не переклад один одного). Це **уніфікується**: текст пишеться один раз (UA) і перекладається на EN автоперекладом, як і решта контенту — без винятку для цієї секції.
